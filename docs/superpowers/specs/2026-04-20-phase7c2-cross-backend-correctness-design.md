# Phase 7-C-2: Cross-Backend Correctness Harness — Design

**Status:** Draft, awaiting user review.
**Date:** 2026-04-20
**Phase predecessor:** 7-C-1 (Real HF Integration + DGX Spark Tier-1, commits `bc503d8` through `6ffe08f`).
**Phase successors:** 7-C-3 (heterogeneous gossip cluster), 7-C-4 (tech-debt cleanup).

## 1. Goal

Establish an empirical agreement bar between `MLXBackend` (Mac, 4-bit Gemma 4 26B A4B-it) and `PyTorchBackend` (DGX Spark, bf16 Gemma 4 26B A4B-it). Phase 7-C-1 validated that each backend independently produces self-consistent output on its own hardware. 7-C-2 closes the remaining "do the two backends actually agree on what the model would say" question — necessary groundwork before 7-C-3 mixes them in a single gossip pipeline.

The bar is **not** bit-exact agreement. 4-bit vs bf16 weight quantization alone guarantees numerical divergence; the goal is to confirm the two backends produce **compatible** output (top-1 agreement on first tokens, top-K overlap across positions) rather than diverging into different distributions.

## 2. Scope and non-goals

**In scope:**
- Unified fixture generator that runs on either backend via `MODEL_SHARD_BACKEND` env var.
- Fixture format captures top-K (K=5) ids + weights per decode position, not just top-1.
- Cross-backend agreement test asserting minimum top-1 match on position 0 and minimum average top-K overlap across all positions.
- MLX and PyTorch fixtures committed to `tests/fixtures/`.
- Side-by-side human-readable report (`tests/fixtures/cross_backend_comparison.md`) regenerated by the test for eyeball diagnostics.

**Explicitly out of scope (deferred):**
- Activation-level `allclose` at layer boundaries — add as follow-up if top-K overlap is weak and we need to localize drift.
- KL / JS divergence metrics on full logit distributions — top-K is sufficient for MVP.
- Tightening agreement thresholds — MVP bars are conservative; tune once we have data.
- Heterogeneous gossip cluster (Phase 7-C-3).
- `Node.py` mlx import gating (Phase 7-C-4 — Task #85).

## 3. Architecture

### 3.1 Unified fixture generator

Replace `scripts/generate_pytorch_tier1_fixture.py` with `scripts/generate_tier1_comparison_fixture.py`:

```python
backend_name = os.environ.get("MODEL_SHARD_BACKEND", "").lower() or "pytorch"
if backend_name == "mlx":
    from model_shard.backends import MLXBackend
    backend = MLXBackend(mlx_lock=<lock>)
    hf_id = "mlx-community/gemma-4-26b-a4b-it-4bit"
elif backend_name == "pytorch":
    from model_shard.backends import PyTorchBackend
    backend = PyTorchBackend(device="cuda")  # or None to auto-detect
    hf_id = "google/gemma-4-26B-A4B-it"
else:
    raise ValueError(f"unknown backend {backend_name!r}")

backend.load(hf_id)
```

The script then runs the same prefill + per-layer atomic decode loop used by `tests/test_pytorch_tier1.py`, but instead of just argmax the **last** logits, it records top-K ids and weights at each decode position. Output fixture file name derives from backend: `mlx_tier1_tokens.json` or `pytorch_tier1_tokens.json`.

Both fixtures use identical JSON shape (see §3.3) so the comparison test is a pure dict-key diff.

### 3.2 Top-K helpers in engine modules

Add a small helper to each engine module so both backends can record top-K consistently:

**`pytorch_engine.py`:**
```python
def top_k_ids_and_weights(
    logits: torch.Tensor, k: int = 5,
) -> tuple[list[int], list[float]]:
    """Extract the top-K token IDs and (softmax) probabilities from the
    last position of a [B, L, V] logits tensor. Returns Python lists for
    fixture serialization."""
    last = logits[0, -1, :]
    weights = torch.softmax(last, dim=-1)
    top_w, top_i = torch.topk(weights, k=k)
    return top_i.cpu().tolist(), [float(w) for w in top_w.cpu().tolist()]
```

**`mlx_engine.py`:**
```python
def top_k_ids_and_weights(
    logits: mx.array, k: int = 5,
) -> tuple[list[int], list[float]]:
    """MLX mirror of pytorch_engine.top_k_ids_and_weights."""
    last = logits[0, -1, :]
    weights = mx.softmax(last, axis=-1)
    top_w, top_i = mx.topk(weights, k=k)  # MLX topk signature — verify
    return [int(x) for x in top_i.tolist()], [float(w) for w in top_w.tolist()]
```

Both accept logits in the same `[B, L, V]` shape that `finalize()` returns. Both return Python lists (JSON-friendly).

### 3.3 Fixture format

Both fixtures share this shape:

```json
{
  "model_id": "<hf_id>",
  "backend": "mlx" | "pytorch",
  "device": "<device>",
  "dtype": "<mlx-4bit> | <bfloat16>",
  "n_positions": 10,
  "top_k_recorded": 5,
  "generator": "<human-readable description>",
  "prompts": [
    {
      "prompt": "The quick brown fox",
      "prompt_ids": [818, 3823, 8864, 37423],
      "top_k_per_position": [
        {
          "ids": [37423, 30234, 29181, 11845, 9827],
          "weights": [0.89, 0.04, 0.02, 0.01, 0.01]
        },
        ...
      ]
    },
    ...
  ]
}
```

`top_k_per_position` is a list of length `n_positions` (10). Each entry is `{ids: [K ints], weights: [K floats]}`. The first entry's `ids[0]` is the top-1 token for position 0 — which is what the test asserts on.

### 3.4 Comparison test

`tests/test_cross_backend_correctness.py` — marked slow (requires both fixtures present) but device-independent (pure JSON diff, runs anywhere).

```python
MIN_FIRST_TOKEN_TOP1_MATCHES = 2   # at least 2 of 3 prompts' position-0 top-1 agrees
MIN_AVERAGE_TOPK_OVERLAP     = 2.0 # top-5 ids average overlap >= 2/5 across all (prompt, position) pairs

@pytest.mark.slow
def test_cross_backend_agreement():
    mlx = json.loads(MLX_FIXTURE.read_text())
    pt = json.loads(PT_FIXTURE.read_text())
    assert [p["prompt"] for p in mlx["prompts"]] == [p["prompt"] for p in pt["prompts"]]

    first_token_matches = sum(
        1 for mp, pp in zip(mlx["prompts"], pt["prompts"], strict=True)
        if mp["top_k_per_position"][0]["ids"][0] == pp["top_k_per_position"][0]["ids"][0]
    )

    overlaps = [
        len(set(mp_pos["ids"]) & set(pp_pos["ids"]))
        for mp, pp in zip(mlx["prompts"], pt["prompts"], strict=True)
        for mp_pos, pp_pos in zip(
            mp["top_k_per_position"], pp["top_k_per_position"], strict=True,
        )
    ]
    avg_overlap = sum(overlaps) / len(overlaps)

    _write_comparison_report(mlx, pt, first_token_matches, avg_overlap, overlaps)

    assert first_token_matches >= MIN_FIRST_TOKEN_TOP1_MATCHES, (
        f"Position-0 top-1 agreement: {first_token_matches}/"
        f"{len(mlx['prompts'])} prompts — below minimum "
        f"{MIN_FIRST_TOKEN_TOP1_MATCHES}"
    )
    assert avg_overlap >= MIN_AVERAGE_TOPK_OVERLAP, (
        f"Average top-5 overlap: {avg_overlap:.2f} across "
        f"{len(overlaps)} positions — below minimum "
        f"{MIN_AVERAGE_TOPK_OVERLAP}"
    )
```

`_write_comparison_report` generates `tests/fixtures/cross_backend_comparison.md` — a side-by-side table of top-K per position for human inspection when the test is interesting (passes or fails).

## 4. Tokenizer equivalence assumption

`mlx-community/gemma-4-26b-a4b-it-4bit` and `google/gemma-4-26B-A4B-it` ostensibly use the same tokenizer. The comparison test asserts `prompt_ids` match across fixtures for the same prompt string — if they don't, the tokenizer differs and the whole comparison is meaningless. Manual verification before Task 3 (Mac MLX fixture generation) against the committed PyTorch fixture is prudent.

## 7. Threshold tuning discipline

The MVP bars (`MIN_FIRST_TOKEN_TOP1_MATCHES=2`, `MIN_AVERAGE_TOPK_OVERLAP=2.0`) are conservative guesses. On first run, expect one of three outcomes:

- **All agree well (bar trivially passed, avg_overlap ≥ 4.0):** tighten bars in a follow-up commit to `MIN_FIRST_TOKEN_TOP1_MATCHES=3` and `MIN_AVERAGE_TOPK_OVERLAP=3.0`. Document the observed numbers in memory.
- **Bars pass with margin (avg_overlap ≈ 2.5–3.5):** keep MVP bars, note observed numbers.
- **Bars fail:** inspect the companion `.md` report. Likely root causes: (a) tokenizer mismatch → Section 4 guard hit; (b) structural bug in one backend; (c) quantization gap larger than expected. Diagnose before adjusting bars downward — loosening a failing test defeats its purpose.

## 8. Decision log

- **D1 — Scope A (offline top-1 + top-K overlap).** Rejected B (activation `allclose`) for MVP because it doubles fixture surface area with uncertain value; layer on later if top-K overlap is weak. Rejected C (online cross-machine) because offline fixtures are reproducible and scp'd easily.
- **D2 — Graded agreement (first-N top-1 match count + top-K overlap).** Rejected hard top-1 agreement because 4-bit vs bf16 precision gap will likely cause divergence within a few positions; rigid bar would fail uninformatively. Rejected KL divergence because we don't need probabilistic precision for MVP.
- **D3 — Top-K = 5.** Rejected K=10 (noisier, top-10 overlap easier to satisfy trivially) and K=3 (too strict, hides signal). K=5 is a standard sweet spot for token-level agreement analysis.
- **D4 — Unified generator script dispatching on env var.** Rejected two separate scripts (duplication) and rejected a single function with both branches (tight coupling). Env var dispatch is how Phase 7-B already chooses backends; reuses the pattern.
- **D5 — Fixture format change for PyTorch side.** The Phase 7-C-1 fixture only records `generated_ids` (top-1 per position). We bump its format to record top-K — the Tier-1 regression test keeps asserting `ids[0]` (= top-1), so no regression intent lost; the extra K-1 entries are additive.
- **D6 — Test is device-independent.** Pure JSON diff. Runs on Mac even though one fixture came from Spark. Simplifies CI.
- **D7 — Conservative MVP bars, threshold tuning as follow-up.** Section 7.
- **D8 — `cross_backend_comparison.md` regenerated every test run.** Side-by-side human-readable diagnostic; commit alongside fixtures.
